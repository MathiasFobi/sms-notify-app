## Codebase Patterns

- **Next.js version:** Scaffold pulled 16.2.9 (the spec said 15). The framework
  conventions (app router, src dir, @/* alias) are unchanged from 15, so the
  story goals are satisfied. If a future story requires strict 15.x behavior
  we can pin with `pnpm add next@15` — note that `--legacy-peer-deps` is not
  needed with pnpm.
- **Single Next.js app, two route surfaces:** `/app` (client portal) and
  `/admin` (operator console) are sibling route segments under
  `src/app/app/` and `src/app/admin/`. Each has its own `layout.tsx` so the
  shell can differ (light for portal, dark for admin). The root `/` is the
  marketing landing page.
- **Tailwind v4 (not v3):** create-next-app installs Tailwind 4 by default.
  CSS lives in `src/app/globals.css` and uses `@import "tailwindcss";` (no
  `tailwind.config.js`). Theme tokens go inside `@theme inline { ... }`.
- **Turbopack is the default builder in Next 16.** First build prints a
  warning if it can find a parent package.json above the project. Pin the
  workspace root in `next.config.ts` to silence it:
  `turbopack: { root: path.resolve(__dirname) }`.
- **Env validation is eager and throws:** `src/lib/env.ts` runs `parseEnv()`
  at module load. Anything that imports `env` (currently `src/db/index.ts`
  and the env test) will crash the process if a required var is missing.
  This is the intended behavior — fail fast on boot.
- **Vitest setup must inject env vars** because `env.ts` reads from
  `process.env` at import time. `vitest.setup.ts` provides placeholders so
  the env test can run. Use bracket access (`process.env["X"]`) — the
  `@types/node` 20+ `process.env` is deeply readonly and rejects dot-access
  assignment.
- **Drizzle + postgres-js singleton:** `src/db/index.ts` stashes the
  postgres-js client on `globalThis` in dev to survive Next.js HMR
  re-imports. `prepare: false` is set so it works against the Neon HTTP
  pooler. Use `pnpm db:generate` to make migrations, `pnpm db:migrate` to
  apply, `pnpm db:push` for local prototyping.
- **Schema conventions:**
  - Primary keys are `serial('id')` for simplicity; can switch to `uuid()`
    later if needed.
  - Foreign keys use `onDelete: 'cascade'` for owned resources
    (accounts, contacts, messages) and `'set null'` for optional
    references (contacts.groupId, messageRecipients.contactId).
  - Append-only ledgers (`creditTransactions`, `webhookEvents`) use
    `uniqueIndex` on provider id to make inserts idempotent.
  - `messageRecipients.phone` is a snapshot of the contact's phone at
    send time, so deleting a contact later doesn't break delivery reports.
  - `inboundMessages.read` is a `boolean` defaulting to `false`, indexed
    on `(userId, read)` so the dashboard unread-count widget is O(1).
- **Route structure:** `src/app/` is the marketing root, `src/app/app/`
  is `/app` portal, `src/app/admin/` is `/admin`. The portal's
  `_actions.ts` lives next to the route group so the portal is
  self-contained.
- **shadcn-style UI primitives are hand-rolled in `src/components/ui/`:**
  No Radix dependency in v1. The contract is:
  - Each primitive lives in its own file (`button.tsx`, `card.tsx`, …)
  - `src/components/ui/index.ts` re-exports them.
  - Server components (e.g. `CreditsBadge`) live in `src/components/`,
    not in `ui/`.
  - Client components declare `"use client"` at the top.
- **Server-action props for client components** that submit a form
  use the `(formData?: FormData) => Promise<void>` shape — accepting
  `?` keeps the prop assignable to both parameterless actions and
  `<form action={...}>` bindings.
- **Server vs client queries:** the portal layout (`src/app/app/layout.tsx`)
  is a server component that does the credit-balance lookup. Pages
  under `/app/*` can issue their own queries (`getDashboardStats` is
  the dashboard's source of truth) without going through the layout.
- **Git workflow:** The repo `sms-notify-app/` is its own standalone git
  repo (`.git` lives at the project root, not at `~/Documents/Workspace/`),
  so it can be pushed to `MathiasFobi/sms-notify-app` independently from
  the other Workspace monorepos.
- **Secrets handling:** `.env.example` carries only placeholders that
  intentionally fail the zod validator (`sk_test_replace_me`,
  `ACreplace_me`) so a missing real value surfaces as a loud error at
  boot. Never inline real keys anywhere in the repo. Builds require the
  real env vars to be present in the shell — CI/deploy must export them.

**Learnings**

- The next-app scaffolder picks the latest Next.js (16 in our case). If
  the spec ever really needs Next 15, pin with `pnpm add next@15
  eslint-config-next@15` after the scaffold and re-run typecheck.
- For multi-segment URLs under src-dir (`/app`, `/admin`), each gets
  its own layout that composes with the root layout automatically. The
  root layout's `<html>` + `<body>` are not duplicated in the segment
  layouts.
- `process.env` writes fail TS strict typecheck in newer `@types/node`.
  Use bracket access in test setup, or cast to a mutable Record. App
  code should never write to `process.env` — only read it via `env`.
- `prepare: false` on postgres-js is the easy win for Neon. The tradeoff
  is no query plan caching, which is fine for an HTTP-pooled serverless
  DB.

**Learnings**

- **lucide-react ships `React.memo(forwardRef(...))` icons.** `typeof
  icon === 'function'` is wrong — it's `"object"`. Test for
  truthiness or `"function" | "object"`.
- **Server actions that take `formData`** need the parameter marked
  optional in the prop type, otherwise the prop can't be assigned
  to a parameterless consumer (e.g. the `<form action={...}>` slot
  on some shadcn primitives).
- **Server component tests use `renderToStaticMarkup` from
  `react-dom/server`.** No jsdom gymnastics needed for
  presentational components.
- **Client components with hooks (`usePathname`) can't be rendered
  outside a Next.js runtime.** Test the data they consume
  (`navItems` array) directly. The "is the active state right"
  check is best covered by a future integration test (Playwright).
- **Tailwind v4 CSS variables need both a `:root` declaration AND
  a `--color-x: var(--x)` line inside `@theme inline`** to be
  consumed as `bg-x` / `text-x` utilities. The shadcn docs assume
  v3; v4 needs the explicit mapping.
- **`pnpm build` requires the env vars in the shell.** The
  placeholder `.env.example` values intentionally fail validation
  to make missing-config obvious at boot. CI must export the real
  values (or use a `.env` that's gitignored).
- **`/app` is the URL surface, not `(portal)`.** Route groups
  (`(portal)`) are parentheses-only and don't appear in the URL.
  The story spec used the (portal) framing as a layout
  group, not as a path segment, so we kept the existing `/app/*`
  structure from US-001/US-002.
