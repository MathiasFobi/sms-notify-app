## Codebase Patterns

The patterns below apply to every story. New developers — read this first.

### Project layout

- **Next.js 16** (Turbopack). `AGENTS.md` warns the version has breaking changes; check `node_modules/next/dist/docs/01-app/03-api-reference/...` before writing any Next-specific code (route handlers, `cookies`, `notFound`, etc.).
- **pnpm** is the package manager (see `pnpm-workspace.yaml`). Use `pnpm run build`, `pnpm run typecheck`, `pnpm test --run`.
- **React 19** + server components by default; client islands must `"use client"`.
- **Tailwind v4** via `@tailwindcss/postcss`. No Tailwind component library — primitives live in `src/components/ui/`.

### Auth seam

- `src/lib/auth/require-user.ts` reads a `user-id` cookie via `next/headers`.
- For tests, `__setCurrentUserIdForTests(id)` overrides the cookie read; `__resetCurrentUserForTests()` clears it. Always pair with `__resetTestDbForTests()` in `beforeEach` / `afterEach`.

### Test DB seam

- `src/test/db.ts` exposes `createTestDb()` (fresh in-memory shim, hermetic per-test) and `getTestDb()` (process singleton).
- Production code goes through `getTestDb()`; unit tests use `createTestDb()` and pass the result into `__<name>Internal` helpers.
- The shim supports `insert` / `update(where = )` / `select(where = )` / `delete`. No `IN` / `JOIN` support — filter and stitch in JS.
- Schema is **duplicated** as raw DDL in `SCHEMA_SQL`. Keep it in sync with `src/db/schema.ts`.

### Action pattern (`__<name>Internal` split)

Every server action file follows the same shape (see `src/lib/actions/inbox.ts`, `src/lib/actions/send.ts`, `src/lib/actions/contacts.ts`):

```ts
"use server";
// Public wrapper — calls requireUser + the internal with getTestDb() + getXxxProvider().
export async function doSomethingAction(args) { ... }
// Test-only internal — accepts a `db: TestDb` and a `provider: XxxProvider`.
export async function __doSomethingInternal(input: { ...; db: TestDb; provider: XxxProvider }) { ... }
```

This lets unit tests drive the internal with a fresh `createTestDb()` and a manually-constructed provider — no singleton coupling, no `next/headers` plumbing.

### Lib-helper pattern (`__<name>Internal` split, no `"use server"`)

Pure read-only helpers that are NOT server actions (e.g. `getDashboardStats`, `getMessageVolume30d`, `getRecentActivity`, `getDeliveryReport`, `getCostSummary` in `src/lib/dashboard.ts` / `src/lib/reports.ts`) follow the same `__<name>Internal` split but live in a regular (non-`"use server"`) module:

```ts
// Public wrapper — no DB arg. Resolves the singleton.
export async function getMessageVolume30d(userId: number): Promise<…> {
  return __getMessageVolume30dInternal({ userId, db: getTestDb(), now: new Date() });
}
// Test-only internal — explicit `db` + `now` (so day-boundary tests
// don't depend on the wall-clock).
export async function __getMessageVolume30dInternal(args: {
  userId: number; db: TestDb; now: Date;
}): Promise<…> { ... }
```

This is the same shape as the action pattern minus `requireUser` / `getXxxProvider()`. Page modules call the public wrapper; unit tests call the `__internal` with `createTestDb()` and a fixed `now` anchor.

### Mock provider convention

- Mock providers (`MockStripeProvider`, `MockSmsProvider`) resolve their backing DB via `getTestDb()` (not via `db/index.ts`). Production code reads `db` from `db/index.ts`; mock-only paths read from `@/test/db`.
- The billing module reads `process.env.BILLING_PROVIDER` **directly** (NOT through `src/lib/env.ts`) so the Stripe secret keys that `env.ts` requires at boot don't gate the mock-only path.
- A `__reset<Provider>ForTests()` helper drops the singleton so the next `getXxxProvider()` constructs a fresh provider.

### Route handler convention

- Handlers live in `src/app/api/<path>/route.ts`. `export const dynamic = "force-dynamic"`.
- Form-urlencoded bodies via `await request.formData()`. JSON responses via `new Response(JSON.stringify(...), { headers: { "Content-Type": "application/json" } })`.
- Dev-only endpoints (e.g. `/dev/webhooks`) gate on `process.env.NODE_ENV === "production"` and short-circuit with `notFound()` from `next/navigation`. Tests flip the gate via a `__set<Name>ProductionOverride(boolean | null)` export.
- Idempotency / replay: every webhook handler is idempotent on its vendor event id (or our synthesized `<sid>:<status>` key for Twilio). Duplicates return `200 OK` with `result: "duplicate"`.

### Page test convention

- Render via `renderToStaticMarkup(await Page())`. The page module is imported dynamically so `next/headers`/`requireUser` deps don't leak into module load.
- Auth override + singleton DB reset are always paired in `beforeEach` / `afterEach`.
- `data-testid`s on every interactive element + on data rows (e.g. `inbox-row-${id}`). Tests assert on those `data-testid`s rather than text content.
- For charts (e.g. SVG bar charts), `data-date` + `data-count` attributes on each shape let tests assert on the rendered series without parsing SVG attributes by position.

### Server-action file: re-export limits

- `"use server"` files in Next.js 16 only allow async functions and type-only exports. Re-exporting schema objects or module-level constants (like `PACKAGES`) breaks the build — `next build` errors out. Import those symbols from their canonical module (`@/db/schema`, `@/lib/billing/packages`) at the call site instead.
- Type-only re-exports (`export type { ... }`) are fine.

### UI primitives

- `Button` (`primary` | `secondary`), `Input`, `Table`, `EmptyState`, `Dialog`, `DialogActions`. All forward refs.
- Server-action forms pair `useTransition` with a hidden action prop. Buttons show "Doing…" while pending. Errors surface in inline `role="alert"` banners.
- There is NO `Card` component — pages render card-shaped `<article>` (or `<section>`) divs directly with `"rounded-lg border border-zinc-200 bg-white p-5"` styling. US-020's dashboard uses this same visual primitive (see `src/app/app/dashboard/page.tsx` → `<article data-testid="dashboard-stat-unread" …>` and `<section data-testid="dashboard-chart-section" …>`).

### Hand-rolled SVG charts

- For small, decorative visualizations (e.g. 30-bar daily volume) the team prefers no chart library. Use a viewBox-based `<svg>` with one `<rect>` per data point + `<text>` ticks. See `src/app/app/dashboard/page.tsx` → `VolumeChart` for the canonical example.
- The page module owns the SVG — no `"use client"`. Server-rendered SVG is fine for static charts and ships zero JS.
- Always include a `<title>` child on each `<rect>` so the bar surfaces a native tooltip on hover.

### Date / time handling

- The test DB's `insert` default for `created_at` and `received_at` is `new Date()` (a `Date` object). Production (Postgres `timestamptz`) round-trips as either string or `Date`. Helpers that read these columns should coerce via `value instanceof Date ? … : Date.parse(value)` rather than relying on a single type.
- For day-bucketed aggregations (e.g. volume series), use UTC `YYYY-MM-DD` keys — locale-free, sortable as strings, stable across timezones. The dashboard helper exposes `formatUtcDay(Date)` and `startOfUtcDay(Date)` for this purpose.

- **Learnings / new patterns:**
  - **`requireAdmin()` belongs at `src/lib/auth.ts` (not under `src/lib/auth/`) per the user-story spec.** Even though `src/lib/auth/` is a folder with an `index.ts` barrel, the spec calls for the flat path. Future specs that pin auth helpers to a specific path should follow the same flat-and-folder co-existence pattern (flat = spec'd public path; folder = actual implementation; index.ts barrel wires them).
  - **Encode audit metadata into `stripe_payment_intent_id` rather than adding a column.** `credit_transactions` has no `note` / `admin_reason` column, so the operator's chosen reason (`support`, `refund`, etc.) is preserved as a `"admin_adjust:<label>"` marker on `stripe_payment_intent_id`. If a future story adds a real `note` column, the marker should move there and the stripe_payment_intent_id prefix should be dropped. The split unlocks three things: (a) the public surface matches the story spec, (b) tests can run the helper against a fresh `createTestDb()` without `__resetTestDbForTests()` + `__setCurrentUserIdForTests()` boilerplate, (c) adding a `now` / `since` knob to a single internal doesn't require a new export for the public wrapper. The dashboard module is now the canonical example of this pattern in a non-`"use server"` lib file.
  - **No `Card` primitive — render card-shaped `<article>` / `<section>` divs directly.** US-020's spec said "use the existing Card primitive" but the project has no `Card` component (only `Button`, `Input`, `Table`, `EmptyState`, `Dialog`, `DialogActions` in `src/components/ui/`). The convention is a `<section>` or `<article>` with `"rounded-lg border border-zinc-200 bg-white p-5"` (with the matching `dark:` variants). When a story mentions a primitive the codebase doesn't yet have, render the existing inline pattern rather than introducing a new component — and call out the missing-primitive gap in the progress notes.
  - **Hand-rolled SVG charts over chart libraries for small, decorative visualizations.** A 30-bar histogram is one `<rect>` per day + sparse `<text>` ticks. A chart library (recharts, visx, chart.js) would add tens of KB for one non-interactive tile. Keep the SVG in the server component — no `"use client"` needed because there's no interactivity (a `<title>` child on each `<rect>` gives a free native tooltip).
  - **Coerce `created_at` / `received_at` defensively.** The in-memory shim's `insert` writes `new Date()`, but real Postgres `timestamptz` may round-trip as either `Date` or ISO string. Helpers reading these columns (e.g. `coerceTimestamp` in `dashboard.ts`) should accept either shape rather than relying on a single type. This bit me on the page test first — the seeded Date was already a Date, so `instanceof Date` short-circuits; but anything fed from `realtime` / PGlite later needs the string-fallback branch.
  - **Sparse x-axis ticks for narrow charts.** 30 text labels in a 600-unit-wide SVG crowds the axis. Tick every 5th day + the final tick keeps it legible (4 labels for 30 bars: days 0, 5, 10, 15, 20, 25, 29). The `<text>` siblings can skip via `null` returns in the map. The `<rect>` chart bars themselves remain unconditional so a screen reader sees the full series even if the tick labels are sparse.
  - **Multi-stream merge = per-stream sort + cap, then merge-sort.** Without `ORDER BY` / `LIMIT` in the shim, fetch each stream's full rowset, sort + slice in JS, concat, final sort by `createdAt` desc. The final desc sort eats the upstream ordering — per-stream sorting is just there to honor the literal cap (e.g. "latest 10 outbound" means newest 10, not arbitrary 10).
